TensorCore — pipeline & parallel execution

SXU pipeline FSM. A single-issue sequencer. FETCH reads one instruction, then branches to the matching EXEC state; dispatch-style ops (MXU) return to FETCH immediately so SXU keeps issuing while MXU runs; collect-style ops (VPU/XLU/LOAD/SELECT) block one extra cycle to capture results from their unit.

SXU fires one instruction per cycle (single-issue); execution units run in parallel for as long as the program doesn't demand their result.

Concurrency rules — what overlaps

  • MXU runs in parallel with SXU. DISPATCH_MXU is issue-only — the Controller latches weight base / activation base / tile length and starts streaming. SXU advances PC immediately; it only synchronizes when a downstream op needs the result, via an explicit WAIT_MXU or via LOAD_MXU_RESULT.
  • VPU collect blocks SXU for 1 cycle (single-cycle ALU) — EXEC_VPU then EXEC_VPU_COLLECT. This is a pipelined 2-stage dispatch, not a stall.
  • Multi-cycle FP reducers stall SXU (SUM/PROD/MAX/MIN tile). EXEC_VPU_COLLECT is guarded on vpu.isDone; while the FpReducer walks the 16-element tile (~16 cycles) SXU is held in the collect state.
  • XLU is single-cycle. Broadcast / transpose ops have the same dispatch + 1-cycle-collect shape as VPU.
  • VMEM read latency = 1 cycle: EXEC_LOAD_REQ then EXEC_LOAD_RESP; VMEM stores are single-cycle.
  • Single-issue, register-renamed by hand. No bypass network — dependencies resolve through the 2-cycle read-after-write observable in the VRegFile. Program must insert an intervening op (or simply not read vd until the cycle after the write) if it wants to observe the result.
  • What does NOT overlap today: two VPU ops; VPU ops against their own prior result (no forwarding); XLU and VPU simultaneously (both go through the same SXU issue slot). Adding a second issue slot or a small scoreboard is the next structural move if we care about beyond-MXU concurrency.

Where each piece lives

  • tinygrad/tinygrad/runtime/ops_tinytpu.py — Python renderer + runtime: UOp → SXU bundle, SXU_PROGRAM executor, sim driver.
  • scripts/tasm.py — textual bundle assembler/disassembler; opcode table mirrored from Python renderer.
  • src/VPU.bsv — 4×4 lane integer + float ALU; hoisted per-column precomputes; shared FpReducer for float reducers.
  • src/FpReducer.bsv — one-adder / one-comparator FSM shared by all float SUM/MAX/MIN/PROD reducers.
  • src/ScalarUnit.bsv — microprogram sequencer; stalls on vpu.isDone for multi-cycle reducers.
  • src/TensorCore.bsv, src/SystolicArray.bsv, src/Controller.bsv — MXU stack + per-core integration.
  • src/XLU.bsv — cross-lane ops (broadcast/transpose/permute).
  • src/VMEM.bsv, src/VRegFile.bsv, src/WeightSRAM.bsv, src/ActivationSRAM.bsv — storage.
  • src/ChipNoC.bsv, src/HBMModel.bsv, src/SparseCore.bsv, src/TinyTPUChip.bsv — chip-level.
  • test/TbTinyTPURuntime.bsv + bdpi/tinytpu_io.c — the bluesim runtime that Python tests dispatch bundles to.
