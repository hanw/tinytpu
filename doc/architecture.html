TinyTPU — Top-Level Datapath & Compilation Flow

4×4 tensor core with shared FP reducer, multi-cycle TranscUnit, 8-bucket PSUM bank, double-buffered Weight/Activation SRAMs + DMA engines, and a dual-issue SXU sequencer (XLU overlaps with non-XLU ops). tinygrad TINYTPU backend lowers UOp graphs to SXU microprograms executed on the BSV runtime. Compilation Flow

TensorCore — pipeline & parallel execution

SXU pipeline FSM. The main sequencer is single-issue with a background XLU collect rule: XLU dispatches advance pc immediately (no XLU_COLLECT state), and a parallel rule writes the result one cycle later while the main FSM fetches/executes the next non-XLU op. MXU dispatch is non-blocking — it latches operand bases into Controller and SXU proceeds. Multi-cycle collectors (VPU reducers, TranscUnit walker) guard on isDone.
Parallel execution timeline. Concurrency on a kernel like MXU-dispatch; VPU-add; XLU-broadcast; VPU-relu; WAIT_MXU; LOAD_MXU_RESULT. Time flows left → right in clock cycles. The main FSM is single-issue; MXU overlaps via dispatch+wait; XLU also overlaps via the background collect rule so its writeback happens while SXU fetches/executes the next op.
Datapath — TensorCore. Storage on the left, engines in the middle, PSUM bucket bank + per-engine result regs on the right feeding VRegFile back. Weight/Activation SRAMs are double-buffered; WeightDMA / ActivationDMA preload the inactive bank in parallel with an MXU dispatch draining the active bank. TranscUnit handles EXP2/LOG2/SIN/COS via a shared multi-cycle walker.

How TinyTPU stacks up against AWS NeuronCore

Both are "matrix + vector + scalar + SIMD engines reading from a shared scratchpad." TinyTPU started with a single shared sequencer and no PSUM bank; recent architectural pushes closed several structural gaps (PSUM bucket bank, dual-issue XLU, output-stationary dataflow mode, double-buffered SRAMs with DMA engines, a transcendental unit). The remaining differences are mostly scale and a sync engine.

Concurrency rules — what overlaps

  • MXU runs in parallel with SXU. DISPATCH_MXU / DISPATCH_MXU_OS are issue-only — the Controller latches operand bases / tile length and starts streaming. SXU advances PC immediately; it synchronizes via WAIT_MXU, LOAD_MXU_RESULT, or a PSUM read.
  • XLU runs in parallel with the main FSM. Each XLU dispatch fires xlu.execute(…), sets xlu_busy+ xlu_dst, advances pc, and returns to FETCH. The next cycle, do_xlu_collect_bg writes vrf[xlu_dst] while the main FSM fetches/executes the next non-XLU op. Structural-hazard guard !xlu_busy blocks back-to-back XLU dispatches.
  • OS-mode MXU accumulator-hold. In DF_OUTPUT_STATIONARY mode, drain skips array.clearAll so consecutive startOS calls accumulate in-place. SXU_MXU_CLEAR starts a fresh epoch; WS dispatches pre-clear.
  • DB SRAM preload-parallel-to-dispatch. Writes target the inactive bank; reads serve the active bank. DMA kick runs concurrently with an ongoing Controller dispatch; an outer swap flips the pointer between tiles.
  • VPU collect blocks SXU for 1 cycle (single-cycle ALU). Multi-cycle FP reducers + TranscUnit walker extend this to N cycles via vpu.isDone guarding EXEC_VPU_COLLECT.
  • PSUM ops are 1-cycle except PSUM_READ (2-step: req + resp). Row-granular variants all single-cycle.
  • VMEM read latency = 1 cycle: EXEC_LOAD_REQ then EXEC_LOAD_RESP; stores are single-cycle.
  • What does NOT overlap today: two VPU ops; a VPU op reading its own prior result (no forwarding); back-to-back XLU ops; VPU + FpReducer / TranscUnit (all share the VPU issue slot).

Instruction encoding — one SXU instr per row, 10 fields
  • 41 SXU opcodes: LOAD_VREG(0) … HALT(7) … DISPATCH_MXU(4), PSUM_*(15–19, 22), DISPATCH_MXU_ACCUMULATE(23) (was the old misnomer MXU_OS), MXU_CLEAR(24), DISPATCH_MXU_OS(25) (real OS), LOAD_MXU_MATRIX_ROW(26), READ_CYCLE(27), LOOP_BEGIN(28)/LOOP_END(29), VZERO(30)/VFILL(31)/VMOV(32), DISPATCH_MXU_OS_ACCUMULATE(33), VNEG(34)/VABS(35), LOAD_LOOP_DEPTH(36), DISPATCH_XLU_ROTATE(37), PSUM_CLEAR_ALL(38), SET_PRED_NE_ZERO(39)/SKIP_IF_NOT_PRED(40)
  • sxu_op=2 (DISPATCH_VPU) indexes into 73 float/int/packed-i8 ALUs + reducers + transcendentals
  • SXU waits on vpu.isDone between VPU dispatches — FpReducer / TranscUnit take N cycles
  • XLU dispatches fire the bg collect rule; structural guard !xlu_busy blocks back-to-back XLU
  • data records (VMEM/WMEM/AMEM) live in a separate section before instrs
  • PSUM bucket idx reuses vmemAddr; vregDst[1:0]=row; vregSrc2[1:0]=psum_mode
  • TASM mnemonics: MXU_OS, MXU_CLEAR, PSUM_*, LOAD_{VPU,XLU}_RESULT

* = multi-cycle via shared FpReducer. Float SUM reductions optionally fuse a post-reduction scalar FMUL (e.g. mean = sum × (1/N)) inside the same kernel. Transcendental VPU ops (EXP2, LOG2, SIN, COS) walk multi-cycle via TranscUnit.

Where each piece lives

  • tinygrad/tinygrad/runtime/ops_tinytpu.py — Python renderer + runtime: UOp → SXU bundle, SXU_PROGRAM executor, sim driver. ~40 pattern-match renderers including self-square / self-cube, swish / silu, softsign, clip / hardtanh, single-bound clamp, leaky_relu, all the transcendentals, plus silent-bug guards.
  • scripts/tasm.py — textual bundle assembler/disassembler; opcode table mirrors the Python renderer and the BSV SxuOpCode enum.
  • src/VPU.bsv — 4×4 lane integer + float ALU; delegates float reductions to FpReducer and transcendentals to TranscUnit.
  • src/FpReducer.bsv — 1 FADD + 1 FMUL + 1 FCMP FSM shared by all float SUM / MAX / MIN / PROD reducers (tile / row / col granularity).
  • src/TranscUnit.bsv — multi-cycle walker implementing EXP2 / LOG2 / SIN / COS via Remez minimax polynomials + range reduction.
  • src/ScalarUnit.bsv — microprogram sequencer; 41 SXU opcodes + background do_xlu_collect_bg dual-issue rule + 1-bit predicate (complete pair: SET_PRED_{IF_ZERO,NE_ZERO} + SKIP_IF_{PRED,NOT_PRED}) + nested-loop stack (depth 4).
  • src/TensorCore.bsv, src/SystolicArray.bsv, src/Controller.bsv — MXU stack + per-core integration. Controller carries a DataflowMode register; startOS preserves accum; clearArray is the explicit reset.
  • src/PSUMBank.bsv — 8-bucket × 4×4 Int32 accumulator with row-granular access. Shared by Controller + PSUM_* SXU opcodes.
  • src/XLU.bsv — cross-lane ops (broadcast / transpose / permute).
  • src/VMEM.bsv, src/VRegFile.bsv — unified scratchpad + engine scratch.
  • src/WeightSRAM.bsv, src/ActivationSRAM.bsv — plain MXU-local operand stores; embedded as .plain sub-interface of the DB variants.
  • src/WeightSRAMDB.bsv, src/ActivationSRAMDB.bsv — double-buffered 2-bank ping-pong wrappers with swap method.
  • src/WeightDMA.bsv, src/ActivationDMA.bsv — minimal DMA stub state machines that stream synthetic tile patterns into the inactive DB bank over N cycles.
  • src/ChipNoC.bsv, src/HBMModel.bsv, src/SparseCore.bsv, src/TinyTPUChip.bsv — chip-level.
  • test/TbTinyTPURuntime.bsv + bdpi/tinytpu_io.c — the bluesim runtime that Python tests dispatch bundles to.
  • BSV TBs: TbCtrlOS, TbCtrlDB, TbCtrlDBDMA, TbWeightDMA, TbActivationDMA, TbTranscUnit, TbSxuPSUM, TbCtrlPSUM — 95 unit tests across all subsystems.
